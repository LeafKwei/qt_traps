# 未将对象移动到线程后就start导致主线程事件循环被阻塞

### 【案例】

首先创建用于描述问题的一些文件：

* work.hpp

```c++
class Work : public QObject{
    Q_OBJECT
signals:
	void to_heartbeat(qint32 i);
    
public slots:
	void run(){
		for(int i = 0; i < 1000; i++){
			QThread::msleep(500);
			emit to_heartbeat(i);
		}
	}
};

class Listener : public QObject{
	Q_OBJECT	
public slots:
    void at_heartbeat(qint32 i){
        qDebug() << "receive heartbeat: " << i;
    }
};
```

* main.cpp

```c++
int main(int argc, char *argv[]){
	QApplication app(argc, argv);
	QThread thread;
	Listener listener;
	Work work;
	
	QObject::connect(&thread, SIGNAL(started()), &work, SLOT(run()));
    QObject::connect(
        &work, SIGNAL(to_heartbeat(qint32)), 
       	&listener, SLOT(at_heartbeat(qint32)),
    	Qt::QueuedConnection  //连接方式必须是Qt::QueuedConnection，原因后面解释
    );
    thread.start();      
    
    return app.exec();
}
```

这段程序的功能很简单，Work在线程中间隔发出`to_heartbeat`的信号，然后Listener在主线程中接收该信号并执行`at_heartbeat`，正常情况下，控制台应该会有规律地打印出`receive heartbeat: xxx`。但实际运行这段程序我们会发现，控制台什么也没有打印。



## 【信号与槽】

这个现象的发生和QT的信号-槽系统以及QThread有关，首先来回顾一下信号-槽系统，我们知道，当使用connect连接信号和槽时，如果我们使用默认的连接方式`Qt::AutoConnection`，那么QT会在**信号发出时**，动态地决定使用哪种连接方式(注意，并不是在connect时就决定)，具体情况如下：

* 如果信号的接收者(receiver)所在的线程和发出信号的线程是同一个线程，则采用`Qt::DirectConnection`
* 其他情况下，采用`Qt::QueuedConnection`

两种连接方式的效果如下：

* `Qt::DirectConnection`：当信号发出时，直接调用槽函数，就像普通的函数调用一样
* `Qt::QueuedConnection`：当信号发出时，在信号的接收者所在的线程产生一个事件，由该线程的事件循环系统调用槽函数

在`Qt::DirectConnection`连接时，哪一个线程发出信号，就由哪一个线程调用槽函数；在`Qt::QueuedConnection`连接时，哪一个对象接收信号，就由该对象所在的线程调用槽函数。前者的对槽函数的调用是同步的，而后者对槽函数的调用则是异步的。



## 【QThread】

接下来再看QThread，在现版本的QT中，我们创建线程一般是按照下方的流程：

1. 创建QThread对象(QThread对象被哪个线程创建就属于哪个线程，并不会自动归属到它所管理的子线程)
2. 创建一个负责执行任务的Work对象，把任务放在一个槽函数中执行(假设函数名字是run)
3. 将Work对象通过`moveToThread`移动到子线程(即QThread)
4. 使用connect函数连接`QThread::started`和`Work::run`
5. 调用start函数

对比上述的内容和开头的例子，我们不难看出在例子中我们忘记调用`moveToThread`函数，所以，Worker对象仍然在主线程而非子线程，进而导致了问题的产生。具体的原因为：在调用`QObject::connect(&thread, SIGNAL(started()), &work, SLOT(run()));`时，由于没有指定连接方式，所以采用`Qt::AutoConnection`连接。那么，当我们调用`thread.start();`后，由于`QThread::started`信号是在子线程发出的，QT将其和Work对象所在的线程一对比，发现发出信号的线程和接收信号的对象不在同一个线程，因此采用了`Qt::QueuedConnection`，向Work对象所在的线程发送一个事件，结果也显而易见，由于Work对象在主线程，那么当然是由主线程去处理该事件并调用`Work::run`函数，而该函数由于是高耗时任务，所以阻塞了主线程，导致主线程的事件循环系统卡死。

接下来再看为什么没有打印出`receive heartbeat`，由于我们使用的是`Qt::QueuedConnection`，那么当`Work::to_heartbeat`信号发出时，就会向主线程发送一个事件，但由于主线程已经被阻塞了，无法处理该事件，自然也就无法调用`Listener::at_heartbeat`。

```c++
QObject::connect(
    &work, SIGNAL(to_heartbeat(qint32)), 
    &listener, SLOT(at_heartbeat(qint32)),
    Qt::QueuedConnection  //连接方式必须是Qt::QueuedConnection，原因后面解释
);
```

(PS: 虽然此处将连接方式改为`Qt::DirectConnection`就可以正常调用`Listener::at_heartbeat`，但并不能解决主线程被阻塞的问题)

明白问题的前因后果后，我们就可以着手解决问题了，只需添加一句`work.moveToThread(&thread)`，将Work对象移动到子线程中，就可以完美解决该问题：

```c++
work.moveToThread(&thread);
QObject::connect(&thread, SIGNAL(started()), &work, SLOT(run()));
QObject::connect(
    &work, SIGNAL(to_heartbeat(qint32)), 
    &listener, SLOT(at_heartbeat(qint32)),
    Qt::QueuedConnection  //连接方式必须是Qt::QueuedConnection，原因后面解释
);
thread.start(); 
```

此时，由于发出信号的线程和接收该信号的Work对象都在子线程，那么Qt就顺理成章地使用`Qt::DirectConnection`连接，从而让Work对象的run函数在子线程中被调用，解决了主线程被阻塞的问题。
