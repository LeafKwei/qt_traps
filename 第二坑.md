# 未将对象移动到线程后就start导致主线程事件循环被阻塞

首先创建用于描述问题的一些文件：

* work.hpp

```c++
class Work : public QObject{
    Q_OBJECT
signals:
	void to_heartbeat(qint32 i);
    
public slots:
	void run(){
		for(int i = 0; i < 1000; i++){
			QThread::msleep(500);
			emit to_heartbeat(i);
		}
	}
};

class Listener : public QObject{
	Q_OBJECT	
public slots:
    void at_heartbeat(qint32 i){
        qDebug() << "receive heartbeat: " << i;
    }
};
```

* main.cpp

```c++
int main(int argc, char *argv[]){
	QApplication app(argc, argv);
	QThread thread;
	Listener listener;
	Work work;
	
	QObject::connect(&thread, SIGNAL(started()), &work, SLOT(run()));
    QObject::connect(
        &work, SIGNAL(to_heartbeat(qint32)), 
       	&listener, SLOT(at_heartbeat(qint32)),
    	Qt::QueuedConnection  //连接方式必须是Qt::QueuedConnection，原因后面解释
    );
    thread.start();      
    
    return app.exec();
}
```

这段程序的功能很简单，Work在线程中间隔发出`to_heartbeat`的信号，然后Listener在主线程中接收该信号并执行`at_heartbeat`，正常情况下，控制台应该会有规律地打印出`receive heartbeat: xxx`。但实际运行这段程序我们会发现，控制台什么也没有打印。

这个现象的发生和QT的事件循环系统以及QThread有关，首先来回顾一下事件循环系统，我们知道，当使用connect连接信号和槽时，如果我们使用默认的连接方式`Qt::AutoConnection`，那么QT会自动根据信号和槽所属对象所在的线程决定连接方式，具体为：

* 信号和槽所属对象在同一线程，使用`Qt::DirectConnection`连接，当信号发出时，直接调用槽函数
* 信号和槽所属对象在不同线程，使用`Qt::QueuedConnection`连接，当信号发出时，在槽所属对象所在的线程产生一个事件，由该线程的事件循环系统调用槽函数

这两种连接方式直接决定了槽函数的执行时机。使用`Qt::DirectConnection`连接时，信号发生的同时就会调用槽函数，此时槽函数的执行时机是确定的；而如果使用`Qt::QueuedConnection`连接时，信号发出时只是产生了一个事件，需要等到槽函数所属对象所在的线程处理该事件时才调用槽函数，此时槽函数的执行时机是不确定的。
